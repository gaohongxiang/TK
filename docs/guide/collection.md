# 数据采集

数据采集用于配合 Codex skill，从 FastMoss 日本站筛选商品，复核后采集到店小秘，并把结果同步到你自己的 Firebase Firestore。

这个模块不是网页上的普通按钮。它由 Codex 在本机执行 `tk-product-selection` skill，使用当前授权采集窗口打开 FastMoss、TikTok Shop 商品页和店小秘。

## 它能做什么

- 按日本站小卖家规则从 FastMoss 找候选商品。
- 过滤合规、物流、品牌/IP、大件重货、高退货等明显不适合商品。
- 对合格商品打开 TikTok Shop 商品页，采集到店小秘采集箱或待编辑商品。
- 把采集记录同步到你的 Firestore `collection_records`。
- 把拒绝品写入轻量去重记录 `collection_excluded_products`，避免同账号重复判断同一商品。
- 记录店小秘编辑进度：采集字段保留 `商品名称`，编辑字段只维护 `店小秘编辑状态`、`编辑时间`、`编辑标题` 和 `编辑判断`，方便追踪编辑结果以及为什么编辑失败或跳过。

## 它不会做什么

- 不绕过平台人工确认、验证码、人机验证、付费确认或发布确认；只保存现场，等待你或你的确认插件处理，页面恢复后继续。
- 不自动发布商品。
- 不修改标题、价格、库存、主图、SKU 或运费模板。
- 不把采集结果上传到本站服务器。
- 不把 FastMoss、店小秘或数据库配置写进 Firestore、Git、采集文件或项目源码。

## 本地配置

Codex 会先检查本机私密配置状态，缺什么才引导你提供什么；已有配置会直接复用：

- 数据库配置：本机没有时提供，本机已有时跳过。
- FastMoss：本机没有可用配置时提供；登录失败、配置失效、会员过期或账号不可用时重新提供。
- 店小秘：按 TK 账号绑定。比如采 `NOMA` 时，如果本机没有 `NOMA` 绑定，就提供 `NOMA` 对应的店小秘账号；已有绑定就自动复用。换新 TK 账号时，如果没有绑定，再提供新的店小秘信息。

这些配置只保存在本机私密配置中，不写入采集表或项目源码。平台人工确认只临时处理，不保存。

## 使用前准备

先确认这些条件已经满足：

1. 网站已经连接你的 Firebase Firestore，Firestore 规则支持商品、订单和数据采集模块。
2. 商品管理或订单管理里已经建立好本次要使用的 TK 账号，例如 `NOMA`。
3. Codex 已安装或能使用 `tk-product-selection` skill。
4. 当前授权采集窗口可用。
5. 店小秘可在同一窗口里采集商品。

如果数据采集页显示数据库权限不足，先按页面提示复制最新 Firestore 规则，到 Firebase Console 发布后刷新页面。

## 日常用法

推荐直接让 Codex 执行：

```text
使用 $tk-product-selection 为 NOMA 账号采集20个符合条件商品
```

如果你没有说账号，Codex 会先问本次同步到哪个 TK 账号。不要根据 FastMoss、TikTok Shop 或店小秘当前登录态猜账号。

## 运行逻辑

采集前只做三类判断：

- 配置：目标 TK 账号、数据库、FastMoss、该 TK 账号绑定的店小秘信息是否齐全。缺什么只问什么，已有配置直接复用。
- 采集窗口：当前授权采集窗口是否可用。不可用才让你处理；可用后普通登录、切号和明确目标店铺选择由 Codex 按本地配置完成。
- 业务：商品是否符合日本站选品硬过滤和评分规则。

标准流程：

1. 确认目标账号，例如 `NOMA`。
2. 检查本地配置，缺数据库配置、FastMoss 或 `NOMA` 的店小秘绑定时才询问并保存。
3. 检查账号存在和采集表读写状态。
4. 创建本批次独立运行目录，并从数据库生成同账号旧商品去重清单。
5. 在当前授权采集窗口里打开 FastMoss，按日本站条件采集店铺和商品。
6. 按规则过滤、评分和去重，生成候选和拒绝文件。
7. 打开 TikTok Shop 商品页，点击店小秘“开始采集”，只放进采集箱或待编辑商品；如果出现平台人工确认，先保存现场并等待页面恢复，超时则写采集失败原因后继续补采。
8. 同步到 Firestore。
9. 打开 TK 网站数据采集页确认结果。

## 内部脚本

这些脚本在 `tk-product-selection` skill 内部，不要求用户安装项目 CLI 或 npm 包：

```bash
node <skill-dir>/scripts/local-credentials.mjs status

node <skill-dir>/scripts/firestore-sync.mjs accounts
node <skill-dir>/scripts/firestore-sync.mjs preflight --account NOMA
node <skill-dir>/scripts/firestore-sync.mjs dedupe data/collection/fastmoss/runs/<run-id> --account NOMA
node <skill-dir>/scripts/firestore-sync.mjs sync data/collection/fastmoss/runs/<run-id> --account NOMA
```

同步脚本直接读取本地私密配置，不依赖项目内 `npm run tk` 或外部 npm CLI。

## 数据采集表里看什么

数据采集页按账号筛选采集记录。主要字段是：

- `账号`：本次采集归属哪个 TK 账号。
- `商品名称`：候选商品名称。
- `选品分`：用于排序的机器评分，不是百分制，也不是最终上架结论。
- `店铺名`、`商品价格`、`商品近7天销量`：FastMoss 采集到的参考数据。`店铺总销售额（日元）` 只用于筛选校验，不在采集表默认展示。
- `采集时间`：同步到采集表的时间，放在采集状态前面。
- `采集状态`：只表示店小秘采集动作结果，只能是已采集或采集失败。候选但尚未尝试采集的商品不能同步进采集表。
- `选品判断`：放在 `采集状态` 后面。成功时写成功选品逻辑；失败时写采集失败原因，不写店小秘编辑失败、跳过或不合格原因。
- `店小秘编辑状态`：商品详情页编辑进度，常见值为未编辑、已编辑、编辑失败。
- `编辑判断`：放在 `店小秘编辑状态` 后面。编辑成功时写编辑结论；编辑失败、跳过或不合格时写决定性原因。
- `商品名称`、`编辑标题`：`商品名称` 是采集时标题，归采集技能维护；`编辑标题` 是店小秘编辑后的标题，归编辑技能维护。
- `编辑时间`：店小秘商品详情页编辑完成后回写的时间。
- `操作`：打开商品链接或复制商品链接。

## 账号筛选和搜索

- 顶部账号标签会先筛选采集记录，再执行搜索。
- 在 `全部` 里，搜索全部账号的数据。
- 在 `NOMA`、`LUMI` 等具体账号里，只搜索当前账号的数据；搜索框不会跨账号标签查数据。

数据采集搜索支持文本和日期组合：

- 裸文本：`NOMA`、`雨衣`、店铺名、`已采集`、`编辑失败`、原因关键词。会搜索账号、商品、店铺、采集状态、编辑状态、选品判断、编辑判断等。
- 裸日期：`05-18` 等于 `采集:2026-05-18`，默认搜索 `采集时间`。
- 定语日期：`采集:05-18`、`编辑:05-25`。
- 英文键盘别名：`cj` 等于采集，`bj` 等于编辑，例如 `bj:05-25`。
- 符号兼容：别名大小写不敏感；支持中文冒号 `：`、`~` 或 `～`、`05/18` 或 `05.18`、全角 `＞＝` / `＜＝`。
- 范围：`采集:05-01～05-18`。
- 比较：`采集:>=05-01`、`编辑:<=05-18`。
- 组合：`NOMA 雨衣 cj:05-01~05-18 bj:>=05-18`。

## 什么时候会停下来

Codex 遇到这些情况会停止并让你处理：

- FastMoss、TikTok 商品页或店小秘出现平台人工确认、验证码或人机验证；Codex 会保存当前商品和页面状态，等待你或确认插件处理完成后继续。
- FastMoss 会员过期、配置失效或账号不可用。
- 店小秘出现付费确认、发布确认。
- 当前授权采集窗口不可用。

普通登录、账号切换和明确目标店铺选择，Codex 会按本地配置代办。

如果平台人工确认长时间没有恢复，当前商品会写入 `采集状态=采集失败`，`选品判断` 写失败原因，例如页面确认等待超时、未确认店小秘采集成功；然后关闭该商品页，继续补采下一个商品。
