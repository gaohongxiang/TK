# 订单页采购统计与帮助信息收纳设计

## 目标

在不改变现有订单页整体视觉风格的前提下，做三件事：

- 增加采购价格统计卡片，显示“当前筛选采购总额”和“全部订单采购总额”
- 把“数据存储说明”从常驻说明块收纳为信息图标 + 弹窗
- 把“最晚到仓”和“订单预警”的公式说明收纳到表头提示图标中

本轮不改 Gist 同步、IndexedDB、本地缓存、账号合并、表格字段顺序、按钮风格，也不做整体页面改版。

## 当前问题

- 订单主面板顶部同时常驻“数据存储说明”和“自动计算公式”，占用了较多竖向空间，导致用户进入页面后核心内容需要继续下滚才能看到。
- 订单列表已经有“采购价格”字段，但页面缺少直观的采购总额概览，无法快速判断当前筛选范围和全部订单一共花了多少钱。
- 公式说明属于低频查看信息，长期常驻不划算；同时移动端没有 hover，不能只按桌面端交互来设计。
- 用户明确要求保留当前页面风格，因此不能把页面重做成新的 dashboard，也不能引入明显不同的视觉语言。

## 方案

### 方案 A：账号标签栏下新增统计卡片区，说明全部收纳

- 账号信息条保留不动。
- 数据存储说明改为顶部信息图标，点击后弹出说明弹窗。
- 公式说明从常驻块改为表头图标提示。
- 账号标签栏下方新增统计卡片区，再往下仍然是现有工具栏和表格。

优点：

- 页面层次最清楚，统计信息有独立位置，但不挤压顶部操作条。
- 进入订单页后，先看到标签和统计，再看到搜索/分页与表格，阅读路径自然。
- 能最大程度保留当前页面风格和既有结构。

缺点：

- 比当前布局多一层卡片区，页面总高度会略有增加。

### 方案 B：统计卡片与搜索/分页工具栏并排

- 统计卡片和搜索、分页、每页条数放在同一视觉层。

优点：

- 页面更紧凑，减少首屏高度。

缺点：

- 顶部信息密度过高，统计与操作混在一起，容易显得拥挤。
- 在手机端更容易堆叠得零碎。

### 方案 C：统计卡片嵌入表格容器上沿

- 把统计卡片直接放进表格容器顶部，作为表格附属信息。

优点：

- 与表格绑定紧。

缺点：

- 统计信息的优先级会被压低，看起来像表格补充说明，不像独立概览。
- 不利于后续再加其他统计项。

## 选型

采用方案 A。

同时确定以下交互口径：

- 统计卡片同时显示“当前筛选采购总额”和“全部订单采购总额”
- “当前筛选采购总额”受账号标签和搜索条件影响，不受分页影响
- 单条订单金额直接累加 `采购价格` 字段，不乘数量
- 桌面端提示信息使用 hover/focus 浮层，移动端点击图标弹出浮层
- 必须保留当前订单页的视觉风格，只做局部结构调整和说明收纳

## 设计

### 页面结构

保留现有订单页主框架：

- 顶部账号信息 / 同步状态 / 操作按钮条保留
- 账号标签栏保留
- 顶部表格工具栏、底部分页工具栏、订单表格保留

替换顶部说明区：

- 删除常驻的 `.ot-storage-note`
- 删除常驻的 `.ot-formula`

新增结构：

- 在同步状态附近增加一个轻量信息图标按钮，用于打开“数据存储说明”弹窗
- 在账号标签栏下方增加统计卡片区，包含两张卡片：
  - `当前筛选采购总额`
  - `全部订单采购总额`

布局规则：

- 桌面端两张卡片左右并排
- 移动端两张卡片上下堆叠
- 统计卡片与现有 `card`、`ot-bar`、`ot-table-toolbar` 视觉语言保持一致，不引入新的重色块、夸张阴影或新字体
- 表格工具栏仍位于统计卡片下方

### 统计口径

#### 当前筛选采购总额

计算范围：

- 受 `activeAccount` 影响
- 受 `searchQuery` 影响
- 不受 `currentPage` 影响
- 不受 `pageSize` 影响

例如：

- 搜索命中 80 条订单，即使当前只显示第 2 页，也统计这 80 条的采购总额
- 切换到某个账号标签后，只统计该账号下且命中搜索条件的订单

#### 全部订单采购总额

计算范围：

- 永远基于 `state.orders` 全量订单计算
- 不受账号标签、搜索、分页影响

#### 单条金额处理规则

- 直接读取订单的 `采购价格`
- 允许空字符串、空值、非法值存在，这些都按 `0` 处理
- 结果不能出现 `NaN`

#### 金额显示规则

- 统一展示为人民币金额格式，例如 `¥ 12,345.67`
- 没有任何订单或筛选无结果时显示 `¥ 0.00`

### 数据存储说明弹窗

#### 入口

- 在顶部同步状态区域旁边放一个信息图标按钮
- 按钮文案保持轻量，不做大按钮；图标应与现有按钮、标签、状态文字风格一致

#### 内容

沿用当前“数据存储说明”的核心含义，仍需覆盖：

- 本地优先：订单先写入浏览器本地数据库（IndexedDB）
- 后台自动同步到用户自己的 GitHub 私有 Gist
- Token 和 Gist ID 需要一起保存，换浏览器或设备时要靠这两项连回原数据
- 云端按账号拆分同步；不同账号互不影响；同账号不同订单自动合并；同一订单冲突时让用户选择保留本地还是云端

#### 交互

- 点击图标打开弹窗
- 点击遮罩、关闭按钮可关闭
- 若现有弹窗样式可复用，则优先复用，不单独发明新弹窗风格

### 表头提示图标

在表头中给以下字段追加信息图标：

- `最晚到仓`
- `订单预警`

#### 最晚到仓提示内容

- `最晚到仓 = 下单时间 + 6 天`

#### 订单预警提示内容

- `订单取消 → 取消订单`
- `已入仓 → 入仓完成`
- `已完成 → 订单完成`
- `已送达 → 订单送达`
- `今天 ≥ 最晚到仓 → 已超期`
- `最晚到仓 - 今天 ≤ 2 天 → 延误风险`
- `最晚到仓 - 今天 > 2 天 → 剩 N 天`

#### 交互规则

- 桌面端：鼠标 hover 或键盘 focus 图标时显示浮层
- 移动端：点击图标显示浮层，点击空白处关闭
- 浮层只承载说明信息，不承担新的业务操作
- 图标和浮层不能干扰表头现有行为；如果某列未来支持排序，排序热点和提示热点需要分离

### 风格约束

本轮实现必须显式遵守以下视觉约束：

- 不改现有页面的整体色彩体系
- 不改现有按钮造型、字号系统、圆角基调
- 不新增与当前订单页风格冲突的面板样式
- 统计卡片看起来应像现有页面自然长出来的模块，而不是另一套产品里的组件
- 收纳说明信息的目标是减少页面占用，而不是让页面“换皮”

## 数据流与代码边界

### 统计数据来源

- 订单原始数据仍来自 `state.orders`
- 当前筛选范围应复用订单列表现有的账号筛选和搜索规则，避免统计口径与表格显示口径不一致
- 统计逻辑应保持为纯计算，不引入新的持久化状态

### 推荐边界

- `js/orders/table.js`
  - 继续承接与“当前列表显示范围”相关的纯逻辑
  - 可补充纯函数，负责基于现有筛选条件导出统计所需订单集合或金额汇总
- `js/orders/index.js`
  - 负责把状态、容器和回调传给订单表格视图模块
  - 负责在渲染时把统计卡片区、帮助入口和表头提示接到页面上

不建议：

- 把统计数据单独塞进全局 `state`
- 在同步模块或 CRUD 模块里维护重复汇总值

## 风险控制

- 统计口径必须和当前筛选逻辑一致，否则用户会质疑卡片金额是否可信
- 移除常驻说明块后，帮助信息不能丢语义，尤其是 Token / Gist ID 和冲突合并说明
- 移动端没有 hover，提示浮层必须有点击路径
- 收纳说明信息时不能破坏当前弹窗、分页、搜索、同步状态等已有交互
- 视觉上只能做局部增强，不能把订单页改成新的大盘样式

## 验证

- Node 级纯函数测试：
  - 全部订单采购总额计算正确
  - 当前筛选采购总额计算正确
  - 当前筛选采购总额不受分页影响
  - 空值和非法采购价格按 `0` 处理
- 视图回归检查：
  - 表头包含 `最晚到仓`、`订单预警` 的提示图标入口
  - 移除常驻数据存储说明块和公式说明块后，订单主面板仍能正常渲染
- 手工验证：
  - 切换账号、输入搜索词、翻页、新增/编辑/删除订单后，统计卡片刷新正确
  - 桌面端可 hover 查看提示
  - 移动端可点击图标查看提示
  - 数据存储说明弹窗可打开和关闭
