# TK 电商工具箱使用文档

本文档面向日常使用者，介绍网站的主要功能、数据连接方式和推荐使用流程。

## 1. 网站包含哪些模块

当前网站分为 3 个模块：

- `利润计算器`
- `商品管理`
- `订单管理`

其中：

- `商品管理` 和 `订单管理` 共用同一个 `Firebase Firestore` 项目
- `利润计算器` 不依赖数据库，但会和订单管理共用一套全局汇率与海外运费参数

## 概览

- 项目已开源：[`https://github.com/gaohongxiang/TK`](https://github.com/gaohongxiang/TK)
- 你可以直接基于开源代码自行部署到 `Cloudflare Pages`
- 利润计算器不依赖数据库，可以直接使用。
- 商品管理和订单管理依赖你自己的 Firebase Firestore 项目，本网站不保存你的数据。
- 第一次使用时，建议先完成数据库连接，再开始录商品和订单。

### 推荐上手顺序

1. 先连接 Firebase
2. 在 `商品管理` 或 `订单管理` 里建立账号都可以
3. 再到 `商品管理` 里录入商品和 SKU
4. 录订单时优先关联商品和 SKU
5. `利润计算器` 不受流程限制，想什么时候用就什么时候用

## 2. 数据库

商品管理和订单管理都依赖你自己的 `Firebase Firestore` 项目。第一次使用时，先把数据库创建好、规则发布好，再回到网站连接。

### 2.1 创建 Firebase 项目

1. 打开 [Firebase Console](https://console.firebase.google.com/)
2. 新建一个项目
3. 添加应用时选择 `网页`
4. 不用勾选 `Firebase Hosting`
5. 复制系统生成的整段 `firebaseConfig`

### 2.2 创建 Firestore 数据库

1. 在 Firebase 项目中进入 `Firestore Database`
2. 创建数据库时建议选择：
   - `区域级`
   - `生产模式`
3. 创建完成后，进入 `Rules` 页面

### 2.3 发布 Firestore 规则

网站提供了最新的 Firestore 规则。你需要把规则复制到 [Firebase Console](https://console.firebase.google.com/) 里并发布。

- 第一次使用时，需要先发布一次规则
- 以后如果页面提示“需要更新 Firestore 规则”，也要重新复制并发布一次

### 2.4 在网站里连接数据库

1. 打开网站
2. 进入 `商品管理` 或 `订单管理`
3. 如果还没连接数据库，页面会出现 `连接 Firebase` 按钮
4. 点击后粘贴整段 `firebaseConfig`
5. 连接成功后，`商品管理` 和 `订单管理` 都会直接复用这一次连接

### 2.5 什么时候需要更新规则

通常不是每次加字段都要重新发布规则。一般在以下情况需要更新：

- 新增了新的集合
- 权限模型发生变化
- 页面弹出了“需要更新 Firestore 规则”的统一提示

如果需要更新规则，直接打开 [Firebase Console](https://console.firebase.google.com/)，复制最新规则，在 `Firestore Rules` 页面重新发布即可。

### 2.6 数据库连接后还有哪些事要知道

- `数据保存在哪里：`订单和商品资料都保存到你自己的 `Firebase Firestore`。网站不会把你的业务数据保存到作者自己的数据库。
- `离线怎么工作：`网站使用 `Firestore 自带的离线缓存`。离线时也能继续操作，恢复联网后会自动同步。
- `换设备怎么恢复：`只要新设备连接的是同一个 Firebase 项目，并填入同一套 `firebaseConfig`，就可以恢复数据。
- `一个项目连一次就够吗：`是。商品管理和订单管理共用同一个 Firestore 项目。连一次，两个模块都会直接复用。

## 3. 利润计算器怎么用

利润计算器用于统一测算售价、利润，以及在已知售价的情况下复盘实际利润。

### 3.1 三种模式

#### 定价旧

适合按旧的计算方式快速反推价格和利润率。

#### 定价新

适合更精细地测算：

- 采购价
- 海外运费
- 平台手续费
- 汇率
- 折扣档位

并支持用内置的 `海外运费计算器` 先算运费，再回填到定价中。

#### 利润复盘

适合在售价已经确定后，复盘实际利润表现。

### 3.2 关键字段口径

- `售价` 录入的是日元
- `采购价`、`海外运费`、`利润结果` 看的是人民币
- 这里的 `汇率` 表示的是：
  - `1 元 = ? 円`

### 3.3 定价新公式与字段说明

- **总费用：** `采购价 + 海外运费`。采购价是 1688 / 拼多多等平台采购商品的实际成本，包含国内运费，平台的减免也算进去。
- **海外运费：** `（基础费 + 每千克重量费 × 计费重 − 用户承担）× 运费倍率 ÷ 汇率 + 贴单费`。可以先用海外运费计算器输入实重、尺寸和货物类型，系统会按计费重估算运费。计算结果会自动填入海外运费框；如果你已经知道运费，也可以直接手动输入海外运费。
  - **基础费 + 每千克重量费 × 计费重：** 官方运费计算公式，具体数值可查看 [日本跨境直邮商家物流服务价卡](https://seller.tiktokglobalshop.com/university/essay?knowledge_id=6411933700818705&role=1&course_type=1&from=search%7BcontentIdParams%7D&identity=1)。
  - **计费重：** 根据实重和尺寸计算出的体积重进行比较，体积重 > 实重 * 1.5，计费重 = 体积重；否则计费重 = 实重。体积大的比较吃亏，除非单价很高。
  - **用户承担：** 不包邮时，用户会出 350 円，在这里扣掉。
  - **运费倍率：** 必须 >= 1，用于整体放大当前运费结果的系数，给运费留出余量。
  - **贴单费：** 中转仓到海外需要重新贴单，默认 1.2 元。
- **日元售价：** `原价 × 折扣 × (1 − 平台手续费)`。商家商品一般会打折扣，目前是按 4 折后还有 1.4 倍来算的，同时 TK 平台会有个手续费，扣完之后才给到商家。
- **利润：** `利润 = 日元售价 ÷ 汇率 - 总费用`
- **原价反推：** `总费用 × 目标利润率 × 汇率 ÷ [基准折扣 × (1 − 平台手续费)]`。系统会根据目标利润率、基准折扣、手续费、总费用和汇率，反推出一个可参考的商品原价，这个价就是上架商品时填入的价格。

### 3.4 汇率和运费参数是否共用

是。

利润计算器中使用的：

- 汇率
- 海外运费参数

会和订单管理共用同一套全局参数。

这意味着：

- 你在利润计算器里调整了汇率
- 订单管理里的日元售价换算和利润汇总也会使用同一套汇率

### 3.5 补充说明

- 海外运费计算时，会同时参考 `实重` 和 `体积重`
- 系统按 `计费重` 来计算运费
- 如果尺寸没有填写完整，系统无法完整校验体积重，结果只能作为粗略参考

## 4. 商品管理怎么用

商品管理用于维护商品主档、SKU等，目的是让后续录订单时可以直接复用，不再重复填写。

### 4.1 商品主档里维护什么

每个商品主档主要包括：

- 所属账号：用来区分商品属于哪个店铺账号，不同账号下可以有同名商品
- TK ID：TikTok Shop 里的商品ID
- 商品名称
- 1688 链接：方便后续回查和采购
- 图片链接
- 货物类型
- SKU 列表：一个商品下面挂一个或多个 SKU

### 4.2 商品和账号是什么关系

商品一定归属于某个账号。

这意味着：

- 不同账号下可以有同名商品
- 商品列表可以按账号切换查看
- 录订单时，系统也会优先按当前账号去筛商品和 SKU

所以录商品时，`账号` 一定要选对。

### 4.3 SKU 架构怎么理解

商品管理不是“一个商品只填一套重量和尺寸”，而是：

- 一个商品主档
- 下面挂一个或多个 SKU

真正影响物流和运费的，是 SKU 这一层。  
也就是说：

- 重量
- 尺寸
- 预估海外运费

最终都按 SKU 来看。

所以：

- 如果一个商品只有一个规格，就维护一个 SKU
- 如果一个商品有多个颜色、尺寸或套装，就维护多个 SKU

### 4.4 新增商品时，推荐怎么录

推荐顺序是：

1. 先填商品主档
2. 再新增 SKU
3. 最后做参数调整

商品主档先填：

- 账号
- TK ID
- 商品名称
- 1688 链接
- 图片链接
- 货物类型

然后再开始录 SKU。

### 4.5 SKU 怎么新增

商品弹窗里有两种方式：

- `新增单个SKU`
- `新增多个SKU`

#### 新增单个 SKU

适合：

- 只有一个规格
- 或者只想手动补一个新 SKU

#### 新增多个 SKU

适合：

- 颜色：白、黑、蓝
- 尺寸：S、M、L

这种组合规格场景。

例如输入：

- 规格维度 1：`白、黑、蓝`
- 规格维度 2：`S、M、L`

系统会自动生成：

- `白 / S`
- `白 / M`
- `白 / L`
- `黑 / S`
- `黑 / M`
- `黑 / L`
- `蓝 / S`
- `蓝 / M`
- `蓝 / L`

### 4.6 SKU 明细

- `SKU 名称`
  - 给人看，建议写成你一眼能认出来的规格名
  - 例如：`白 / S`、`黑 / L`
- `SKU ID`
  - 系统会自动生成
  - 主要用于内部识别和订单关联
- `重量`
  - 填商品这个 SKU 的实际重量，单位是 `g`
- `尺寸`
  - 按 `长×宽×高` 填写，单位是 `cm`
  - 例如：`20×15×10`

### 4.7 参数调整怎么用

这里有两种典型情况：

- `所有 SKU 的重量和尺寸都一样：`先生成全部 SKU，`匹配关键词` 留空，直接填写重量和尺寸，系统会同步到全部 SKU。
- `按分组不同：`例如按 `S / M / L` 分组，先生成全部 SKU，`匹配关键词填S` ，同时填写重量和尺寸，系统会同步到所有匹配到S的SKU，如白 / S，黑 / S，蓝 / S，没命中的 SKU原来的值保持不变。其他同理。
- `各SKU的重量尺寸输入框也可单独修改：`比如其中某一两个 SKU 需要单独修正。

### 4.8 参数调整的联动规则

- `匹配关键词` 留空：输入重量和尺寸后，会同步到全部 SKU。
- 填了 `匹配关键词`：只会同步到命中的 SKU。
- 没命中的 SKU：原来的值保持不变。
- 如果你直接改某一行 SKU 自己的重量和尺寸：就按这一行自己的值为准，不会再被其他不相关的批量调整清空。

### 4.9 保存时会校验什么

保存商品时：

- 每个商品至少要有一个 SKU
- 每个 SKU 都必须有 `SKU 名称`
- 每个 SKU 都必须有有效的 `重量` 和 `尺寸`

如果缺少这些值：

- 对应输入框会变红
- 系统不会让你保存

### 4.10 商品列表怎么看多 SKU

商品列表支持：

- 按账号查看
- 搜索
- 分页
- 点击行展开多 SKU 明细

当某个商品有多个 SKU 时，点击该商品所在行即可展开，查看每个 SKU 的：

- SKU 名称
- SKU ID
- 重量
- 尺寸
- 预估海外运费

### 4.11 商品和订单管理怎么联动

商品管理不是单独存在的，它和订单管理是联动的。

在订单管理里：

- 新增订单时可以关联商品，以及继续选到具体的 SKU。
- `关联商品` 下拉支持按 `商品 ID / 商品名称` 搜索。
- `关联 SKU` 下拉支持按 `SKU ID / SKU 名称` 搜索。

关联后，系统会自动带出：

- 商品名称
- 单件重量
- 单件尺寸

而且订单保存时会写入自己的快照，所以后面即使你改了商品或 SKU，历史订单也不会被反向改乱。

### 4.12 1688 链接怎么操作

商品列表里的 `1688` 提供两个按钮：

- `打开`
- `复制`

## 5. 订单管理怎么用

订单管理用于记录采购、物流、入仓进度，并汇总销售额、支出和预估利润。

现在一张订单可以包含：

- 多个商品
- 多个 SKU
- 每个 SKU 各自的数量

也就是说，订单已经不是“一个订单只对应一个商品 / 一个 SKU”的结构了。

### 5.1 订单管理包含什么

订单管理包含：

- 账号标签
- 订单列表
- 搜索、分页和导出
- 新增 / 编辑订单
- 订单明细
- 统计汇总

### 5.2 订单和账号是什么关系

每条订单都归属于某个账号。

这意味着：

- 录订单时，先选账号
- 切换账号标签后，列表会按账号查看
- 关联商品和 SKU 时，也会优先按当前账号去筛

### 5.3 新增订单时，推荐怎么录

推荐顺序是：

1. **先填基本信息**
   - `账号`
   - `下单时间`
   - `订单号`
   - `采购日期`
   - `最晚到仓时间` 会按下单时间 + 6 天自动生成

2. **再填订单明细项**
   - 在 `订单明细` 里逐条添加商品 / SKU / 数量
   - 如果商品资料已经建好，优先关联商品和 SKU
   - 关联后，系统会自动带出：
     - 商品名称
     - 单件重量
     - 单件尺寸
   - 这些自动带出的值，后面仍然可以手动修改

3. **再看订单级汇总**
   - `总件数`：按所有明细数量自动求和
   - `总重量`：按所有明细 `单件重量 × 数量` 自动汇总
   - `总尺寸`：不会自动乱算，需要你按实际打包情况自己调整
   - `预估总海外运费`：会按当前的 `总重量 + 总尺寸` 自动计算

4. **再填订单级金额**
   - `总售价（日元）` 是整张订单的总售价
   - `总采购额（元）` 是整张订单的总采购额
   - 它们不再按明细逐条填写

5. **最后补快递信息**
   - `快递公司`
   - `快递单号`
   - `订单状态`
   - `订单预警` 会自动生成

关联商品不是强制项，但推荐这样录，后面会顺很多。即使不关联商品，也可以手动把订单录完整。

### 5.4 关联商品和 SKU 后，会发生什么

在新增或编辑订单时：

- `关联商品` 下拉支持按 `商品 ID / 商品名称` 搜索
- `关联 SKU` 下拉支持按 `SKU ID / SKU 名称` 搜索
- 选中商品和 SKU 后，系统会自动带出：
  - 商品名称
  - 单件重量
  - 单件尺寸
- 这些自动带出的值，后面仍然可以手动修改，不会被锁住。

订单保存时会写入一份自己的快照，所以：

- 后续就算商品资料变了
- 历史订单也不会被反向改掉

### 5.5 订单明细怎么理解

`订单明细` 是订单里真正买了什么。

一张订单里可以有多条明细，每条明细代表：

- 一个商品
- 一个 SKU
- 这个 SKU 的数量

这意味着：

- 一个订单可以有多个 SKU
- 同一个 SKU 可以通过 `数量` 表示买了多件

通常建议：

- 同一个 SKU 直接用 `数量` 表示多件
- 不要重复加两条完全一样的明细

### 5.6 重量、尺寸和预估总海外运费怎么理解

现在重量、尺寸和运费分成了两层：

- `订单明细` 里的 `单件重量 / 单件尺寸`
- `订单级` 的 `总重量 / 总尺寸 / 预估总海外运费`

规则是：

- `总件数`：按所有明细数量自动求和
- `总重量`：按所有明细 `单件重量 × 数量` 自动汇总
- `总尺寸`：不会自动乱算，需要你按实际打包情况自己调整
- `预估总海外运费`：会按订单级最终的 `总重量 + 总尺寸` 自动计算

所以：

- 重量系统可以给你自动参考
- 尺寸仍然建议你按实际包裹自己确认
- 预估总海外运费会始终跟着当前的总重量和总尺寸重新计算
- 如果你改了总重量或总尺寸，预估总海外运费也会跟着更新

### 5.7 订单里的金额口径

订单管理中金额口径如下：

- `总售价`：录入的是 `日元`
- `总采购额`：录入的是 `人民币`
- `预估总海外运费`：录入的是 `人民币`
- `预估利润`：显示的是 `人民币`

预估利润公式：

`预估利润 = 总售价（日元）÷ 汇率 -（总采购额 + 预估总海外运费）`

其中汇率使用的是和利润计算器共用的全局汇率。

### 5.8 订单状态会自动变化吗

会，但只在合适的场景下自动调整。

- 新增订单时，如果录入了快递单号，系统会按规则自动把订单状态切到更合适的状态
- 如果快递公司没填，系统也会尽量根据单号自动识别
- 如果你后续手动修改状态，系统不会强行覆盖你的手动选择

### 5.9 订单统计怎么看

订单页顶部有两组统计：

- `当前筛选`
- `全部订单`

每组会展示：

- 总销售额
- 总采购额
- 预估总海外运费
- 预估总利润

说明：

- `总销售额` 是收入
- `总采购额 + 预估总海外运费` 是支出
- `预估总利润 = 总销售额 - 总支出`

### 5.10 搜索、分页和导出怎么用

订单管理支持：

- 搜索：下单时间、订单号、产品名称、快递信息
- 分页：在列表上方和下方切换
- 导出：`导出 CSV` 会把当前列表里的订单导出成表格文件
- 新增订单后，页面会保留在当前分页，不会跳回第一页

如果你输入的是日期格式，系统会优先按 `下单时间` 搜索。

## 6. 常见问题

### 6.1 为什么商品管理和订单管理都能直接用同一个数据库

因为这两个模块本来就是共用同一个 Firestore 项目。  
只要连接一次，两个模块都会直接复用。

### 6.2 为什么商品里要先做 SKU，再去订单里选 SKU

因为真正影响运费的，通常不是商品主标题，而是具体规格。  
不同 SKU 可能在以下方面不同：

- 重量
- 尺寸
- 预估海外运费

所以订单里应该选具体 SKU，而不是只选商品主档。

### 6.3 订单和商品为什么都还能离线用

因为网站使用的是 Firestore 自带离线缓存。  
即使暂时离线，也可以继续操作；恢复网络后会再同步。
