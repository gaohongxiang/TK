# 出海匠日本站选品标准流程

这个流程使用当前已连接的 Chrome 会话完成出海匠网页采集。出海匠、TikTok 商品页、店小秘和采集表同步都必须在同一个会话里完成。

## 目标

从出海匠 TikTok 日本站商品页筛选适合小卖家的商品。出海匠是默认优先来源，具体找货方式由 `references/rules.md` 的统一策略决定：

1. 关键词搜索、页面筛选、类目榜单、爆品延伸、店铺挖掘、相似商品或种子链接扩展
2. 商品详情页取得 TikTok 官方商品链接
3. 按 `rules.md` 的全局规则和当前策略完成硬过滤和评分
4. 打开 TikTok 商品页，点击店小秘“开始采集”
5. 生成采集记录并同步到采集表

出海匠不是 FastMoss API，也不是导出优先流程。当前可用路径是网页 DOM 采集：列表拿候选，详情拿 TikTok 官方链接，再去 TikTok 商品页触发店小秘插件。

## 会员状态和登录

出海匠是会员来源，采集前必须检查页面会员状态和数据权限。

- 页面已经登录且可查看日本站商品、店铺、近 7 天销量、销售额和 TikTok 官方链接时，沿用当前 Chrome 会话，不重新登录。
- 页面处于游客、未登录、会员过期、权限不足、数据被遮挡、跳转登录页或提示升级/续费时，才使用本地私密配置里的出海匠账号自动登录。
- 本地配置用 `local-credentials.mjs set-chuhaijiang --username <账号> --password <密码>` 保存；`status` 里应显示 `hasChuhaijiangDefault=true` 且 `chuhaijiangStatus=active`。
- 如果自动登录失败、页面仍显示过期/权限不足，或本地状态已是 `expired`，运行 `local-credentials.mjs mark-chuhaijiang-expired --reason <原因>` 标记失效，并提醒用户提供新的出海匠账号密码。
- 用户提供新账号密码后，先保存到私密配置，再在当前 Chrome 会话登录并重新检查会员状态；确认页面会员可用后继续采集。
- 不复制出海匠 cookie、LocalStorage、IndexedDB；只在当前已授权 Chrome 会话里登录和检查。

## TikTok 官方链接和店小秘插件

出海匠详情页里的 `TikTok 官方链接` 是店小秘插件采集的正式入口。执行时必须打开这个真实 TikTok 商品链接，确认页面已经进入商品详情并出现店小秘插件条，再点击插件里的“开始采集”。

为降低平台确认触发率，优先从出海匠详情页点击官方链接或按页面自然跳转进入 TikTok 商品页；必须直接打开链接时，只打开一个商品页，等待页面稳定后再操作。不要批量 `goto` TikTok 链接、不要并发开商品页、不要连续快速打开后立即关闭。每个商品之间保留 20-60 秒间隔；同一批次触发过验证后，后续商品按更慢节奏执行。

不要用店小秘后台的“链接采集”替代这个步骤。TikTok 平台在店小秘后台“链接采集”里可能返回“需使用插件采集”，这只说明后台链接采集路径不可用，不能作为该商品采集失败的结论。只有真实 TikTok 商品页打开后插件点击结果，才是 `采集状态` 的依据。

判断规则：

- TikTok 商品页正常打开且插件提示“采集成功”或“前往采集箱”：写 `采集状态=已采集`。
- TikTok 商品页出现验证页、人机确认、滑块拼图、`Security Check`、`Verify to continue` 或空白页：保持当前商品页可见，写 `page_confirmation_state.json`，并调用 `scripts/notify-human-verification.mjs` 播放一次提示音，等待用户或用户确认插件处理；恢复后继续点店小秘插件。
- 等待超时仍未恢复商品详情：不要把页面立即关到用户看不到；先保留现场并记录原因。用户没有要求继续等时，写 `采集状态=采集失败`，失败原因写“已打开真实 TikTok 商品链接，但商品页触发滑块拼图/人机验证，等待超时未恢复，未点击店小秘采集。”
- 已点击店小秘“开始采集”但没有看到成功提示：先检查页面顶部绿色提示、底部插件按钮状态和采集箱入口；仍无法确认时才写 `采集状态=采集失败`。
- 每个商品完成或放弃后关闭该 TikTok 商品页，避免页面越开越多。

## 当前验证结果

2026-05-29 小批量测试已跑通：

- 出海匠列表可读商品、店铺、价格、近 7 天销量、销售额、类目。
- 出海匠详情页可取得 TikTok 官方商品链接。
- TikTok 商品页能显示店小秘“开始采集”按钮。
- 3 个测试商品点击后页面显示“采集成功”。

测试目录示例：

```text
data/collection/chuhaijiang/runs/20260529-223131-chuhaijiang-test/
```

正式采集仍必须走新运行目录和数据库去重，测试商品不能自动算作新批次。

## 出海匠来源硬门槛

- 国家/地区：日本。
- 商品状态：在售。
- 商品近 7 天销量：大于 0。
- 商品详情页必须取得真实 TikTok 商品链接。
- 商品必须通过 `references/rules.md` 的官方红线和运营红线。
- 默认策略不套用店铺总销售额 10,000-80,000 日元门槛；这个区间属于 `small-shop-filter` 策略。用户明确要求“小店条件/原 FastMoss 条件”时，出海匠也要补齐店铺总销售额和店铺近 7 天销量，补不到就不计入本批次。

## 入口策略

出海匠不是只能关键词采集。可用入口包括：

- `mixed-discovery`：默认策略，组合轻小类目、页面筛选、少量关键词、同类/店铺线索。
- `keyword`：按日文关键词搜索。
- `category-ranking`：按类目榜单或销量榜找线索。
- `hot-extension`：按爆品拆需求、卖点和场景，再找低风险延伸款。
- `shop-mining`：从合格商品或低风险店铺继续挖同店铺商品。
- `similar-product`：找相似商品、替代款、配件款、耗材款。
- `seed-link`：用户给指定链接或种子商品时，先拆解再决定采集或延伸。

不要从默认销量榜直接采。默认榜常混入服装、食品、带电产品、眼镜和大件，过滤成本高。

关键词策略下，优先搜索轻小、可压缩、无电、非入口接触的日文关键词：

- `トラベルポーチ`
- `収納ポーチ`
- `防塵カバー`
- `洗濯ネット`
- `ゴミ取りネット`
- `マイクロファイバー`
- `収納袋`
- `掃除クロス`
- `ケーブルホルダー`
- `コードホルダー`
- `家具脚カバー`
- `シリコンブラシ`

所有入口结果都必须再过硬过滤。不要因为关键词、榜单、爆品或同店铺看起来安全就直接采。

## 出海匠特别拒绝词

除 `rules.md` 外，出海匠页面常见以下日文/中文词，命中时直接拒绝或强复核：

- 眼镜：`サングラス`、`眼科`、`偏光`、`調光`、`UV400`、`レンズ`、`眼鏡`。
- 带电/带磁：`USB充電`、`内蔵バッテリー`、`電動`、`LED`、`コンセント`、`マグネット`、`磁気`。
- 首饰/吊饰：`チャーム`、`パール`、`ビジュー`、`キスロック`、`キーリング`、`キーホルダー`、`手机吊饰`。
- 大件/硬件：`ラック`、`シューズラック`、`物干し`、`椅子`、`チェア`、`フラフープ`、`1.5kg`。
- 工具/尖锐：`除草`、`雑草`、`園芸工具`、`刃`、`レーキ`、`スクレーパー`。
- 汽车：`洗車`、`車用`、`車載`、`バイク`。汽车相关默认拒绝，除非用户明确放宽。
- 美妆/身体护理：`フェイス`、`顔マッサージ`、`かっさ`、`美容`、`メイク`、`化粧`。
- 食品/入口接触：`米`、`食品`、`水筒`、`ボトル`、`ストロー`、`カップ`。

## 标准流程

1. 读取 `references/rules.md` 和本文件。
2. 运行 `local-credentials.mjs status`，检查出海匠配置和状态；明确目标账号后运行 `firestore-sync.mjs preflight --account <账号名>`。
3. 在当前 Chrome 会话打开出海匠并检查会员状态；页面会员可用就沿用，页面过期/未登录/权限不足时按“会员状态和登录”处理。
4. 创建新目录 `data/collection/chuhaijiang/runs/<run-id>/`。
5. 运行 `firestore-sync.mjs dedupe data/collection/chuhaijiang/runs/<run-id> --account <账号名>`，只用数据库生成去重清单。
6. 在当前 Chrome 会话打开：

```text
https://www.chuhaijiang.com/app/discover/tiktok/products?country=JP
```

7. 按本批次策略找候选，把列表结果保存为 `products.json`。每条尽量包含：
   - 商品 ID、商品名、出海匠商品链接
   - 店铺名、出海匠店铺链接
   - 类目、价格、近 7 天销量、近 7 天销售额、总销售额
   - 原始行文本
8. 对初筛通过的候选逐个打开出海匠详情页，取得 `tiktokOfficialUrl`，并补充详情字段。没有 TikTok 官方链接的商品拒绝。
9. 运行筛选脚本，并按本批次策略传入 `--strategy`：

```bash
node <skill-dir>/scripts/select-chuhaijiang-products.mjs data/collection/chuhaijiang/runs/<run-id>/products.json --account NOMA --strategy mixed-discovery
```

10. 可运行 `init-dianxiaomi-status.mjs` 做字段预检，但最终 `collection_records.csv` 必须由 RPA attempts 生成，不能把未尝试候选直接写入采集表。
11. 运行 `run-dxm-collection-rpa.mjs <run-dir> --account NOMA --target-successes 20`。Runner 按固定状态机逐个从候选记录打开真实 TikTok 商品链接，确认商品详情和店小秘插件条出现后，点击店小秘“开始采集”。不要用店小秘后台“链接采集”替代插件采集。每完成或放弃一个商品都关闭该商品页；遇到验证时先保留页面等待恢复，再决定是否失败。
12. 点击后页面显示“采集成功”或“前往采集箱”才写 `采集状态=已采集`；出现重复时点击取消并按已采集记录；出现验证、登录异常、等待超时或无法确认时写 `采集状态=采集失败`，并在 `选品判断` 写失败原因。用户说“继续”时用 `--continue-blocked --retry-verification` 从 `page_confirmation_state.json` 的同一商品继续。
13. 同步前复核 `商品近7天销量` 非空且大于 0、`商品链接` 为 TikTok 商品链接、`是否旧商品=否`、`是否计入本次=是`。
14. 运行 `firestore-sync.mjs sync data/collection/chuhaijiang/runs/<run-id> --account NOMA`。

## 页面人工确认

出海匠、TikTok 或店小秘出现验证码、人机验证、风险确认、登录失效、付费确认或发布确认时，按主 `SKILL.md` 的“平台人工确认”执行。不要接入第三方打码，不要绕过；保持当前页，等待用户或用户确认插件处理后继续。

出海匠来源页触发验证或会员/权限遮挡时，如果还没有进入具体 TikTok 商品页，不写 `collection_records`，只保留当前筛选状态和已确认候选。页面恢复后继续同一关键词、榜单、店铺或详情页；同一来源连续两次验证且没有新增候选时，暂停本批次来源采集并汇报页面状态。

## 输出文件

出海匠运行目录应包含：

- `products.json`：页面采集和详情补全后的商品数据。
- `dedupe_manifest.json`：数据库来源的同账号旧商品去重清单。
- `selection_candidates.json/csv`：候选商品过程文件。
- `selection_rejects.json/csv`：拒绝品过程文件。
- `rpa_state.json`：当前商品、阶段、恢复点、目标数量、成功数和失败数。
- `rpa_events.jsonl`：RPA 每一步业务事件日志。
- `collection_attempts.json`：已尝试店小秘采集的结构化结果。
- `collection_records.csv`：只包含已尝试店小秘采集的商品。
- `page_confirmation_state.json`：只有遇到平台人工确认时才需要。

`collection_records.csv` 仍只写采集字段，编辑字段由 `tk-product-editor` 处理：

```text
账号, 商品ID, 商品名称, 店铺名, 商品价格, 商品近7天销量, 采集时间, 采集状态, 选品判断, 商品链接, 出海匠链接, 店铺链接
```

采集阶段不写 `店小秘编辑状态`、`编辑时间`、`编辑标题`、`编辑判断`。
